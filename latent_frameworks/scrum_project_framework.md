# Latent Framework でみた Scrum の構造と限界、及びその補正

---

## 1. 概要

Scrum 開発手法を、Latent Framework で構文かしてみる。

| 視点                   | **Scrum が強い層**                              | **Scrum が弱い／空白になりやすい層**                                                                  |
| -------------------- | ------------------------------------------- | ---------------------------------------------------------------------------------------- |
| **Latent Framework** | 🧵 **Craft**（作業構築）<br>🥼 **Science**（計測・検証） | 📕 **Story**（衝動 / 論理 / 転調）<br>🌾 **Economy**（流通・資本）<br>⚔️ **Force** の一部（Norm・Propaganda） |

### 要点

* **Scrum は「作る」🧵 と「測る」🥼 に特化した軽量エンジン**。
  Story（動機・意味）や外部接続（Economy／Force）は **“あるもの” として前提**、あるいは **外付けモジュール** として扱う設計。
* 企業内で導入される多くのチームは十分な **👏Blues（痛みを伴う動機）** を共有しておらず、**“やらされ 🎭 Scrum”** に陥りやすい——これはフレームの限界ではなく目的の範囲外である。そのため、補完が必要である。

## 3. 限界と “Marionette” リスク

| 観察点             | 典型症状                        | 構文的原因 (Latent)                 |
| --------------- | --------------------------- | ------------------------------ |
| **Story 希薄**    | Sprint Goal が「チケット合算」       | 👏 Blues を注ぐ儀式無し               |
| **儀式の作業化**      | Daily＝進捗報告 / Retro＝KPT 箇条書き | ☝️ Logic / 🫱 Reframe 再確認ゼロ    |
| **統合遅延 (スケール)** | 依存隠蔽 → リリース遅延               | 🔗 Control + 🖥 Information 不足 |
| **外部と断絶**       | 法務・資金・宣伝が孤立                 | 🌾 Economy / ⚔️ Force 未扱い      |

### 3.1 🎭 Marionette（傀儡 Story）

| 状態  | Latent 的構造           | 現場例                     |
| --- | -------------------- | ----------------------- |
| 傀儡化 | Blues 欠如＋外部 Logic 依存 | PO が伝書鳩化／Goal が経営者の思いつき |

### チェック３問

1. *この目標は「誰」の語り？* 2) *成功で一番喜ぶのは？* 3) *成否をユーザー行動で語れる？*
   １つでも **No** → 傀儡化の疑い。

## 4. Story 注入 & 外付け補完

### 4.1 Story 注入リチュアル

| タイミング         | 施策例                              | 補完要素       |
| ------------- | -------------------------------- | ---------- |
| **Day 0**     | “Why” ワーク（痛み３段掘り）                | 👏 Blues   |
| **Sprint 開始** | Goal＝「問い(Blues)+価値根拠(Logic)」２行宣言 | ☝️ Logic   |
| **Daily 冒頭**  | 「昨日の学び／今日の仮説」１行共有                | 🫱 Reframe |

### 4.2 Economy / Force 外付け

| モジュール       | ねらい                        | 実装ヒント            |
| ----------- | -------------------------- | ---------------- |
| 資金・リスクボード   | Capital 可視化                | CFO/スポンサーと週次     |
| リリース横串カレンダー | Flow (Logistics+Market) 一望 | SoS で１枚ガント       |
| 規範チェックリスト   | Law / Custom 逸脱検知          | Retro「Done?」欄に組込 |

### 4.3 SoS 拡張（Control / Information）

| ルール                 | 目的      | Latent                      |
| ------------------- | ------- | --------------------------- |
| SoS代表に**決裁権**       | ブロッカー即断 | 🗝 Authority                |
| 冒頭30秒 **Goal再読**    | ゴールずれ防止 | ☝️ Logic                    |
| 依存タスク **48h オーナ指名** | 遅延連鎖防止  | 🔗 Control + 🖥 Information |

## 5. 各 Scrum 要素の日本語リネーム ― “合わせる”感覚を外す Reframe

Scrum を行うときに、与えられた名称をそのまま用いることが多い。だがその場合、「Scrum を行うこと」が目的に Story が変容することがある。

それを防ぐ 🫱 Reframe の提案として、各構成要素を母国語にすることを行なってみる。

### 翻訳例

| 種別       | 英語名                  | 提案する日本語     | 目的（合わす→語る）    |
| -------- | -------------------- | ----------- | ------------- |
| ロール      | Product Owner        | **語りの番人** | “所有者”より語り手を強調 |
|          | Scrum Master         | **守り人**   | 支配者イメージを除去    |
|          | Developers           | **仕立て人**  | 職種より行為を強調     |
| イベント     | Sprint Planning      | **段取り決め**   | 手順ではなく設計行為    |
|          | Daily Scrum          | **今日のひと言**  | 監視→学びシェア      |
|          | Sprint Review        | **発表会**  | 審査会→お披露目      |
|          | Sprint Retrospective | **ふりかえり**   | 作業KPT→問いの改善   |
| アーティファクト | Product Backlog      | **やりたいこと一覧**  | -      |
|          | Sprint Backlog       | **やること一覧**  | -    |
|          | Increment            | **お届け物**     | 市場へ送る価値梱包     |

### 効果
- *英語ラベル = “外から合わせる手順”* という刷り込みを外し、
- **行為そのもの／語りそのもの** に焦点を再配置 → Blues が感じやすくなる。

実際のチームの色に合うものをそれぞれ選ぶのが良いと思われる。

---

## 6. まとめ

1. **Scrum は Craft＋Science 専門エンジン**。
2. **Story／Economy／Force を意図的に補わなければ “やらされ化”**。
3. **Latent Framework** を「前段ブースター／空欄チェッカー」として

   * Blues・Logic・Reframe を可視化し、
   * 資金・規範・情報統制を外付け。
4. **名称リネーム (日本語化) + Story 注入儀式** で「合わせる手順」→「自ら語る行為」へ Reframe。

> **Scrum の失速は“フレームの限界”ではなく “補完忘れ”**
> —— Story-Driven × Science-Based の循環を取り戻そう。
