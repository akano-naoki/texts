# Coding Agent と協業する Ruby on Rails に乗ったドメイン駆動開発
> Railsの価値を「統合と規約」と捉え直し、ActiveRecord中心主義（とその副作用）から距離を取りつつ、  
> Coding Agent（Codex等）と共存できる“現実的な開発作法”を整理する。

---

## 0. この文章で扱うこと
- Railsが「なぜ立ち上げに強いのか」
- Railsが「なぜfat modelに陥りやすいのか」
- SOA / マイクロサービス / モジュラーモノリスの誤解されやすい境界
- ActiveRecordの“兼務”が生む設計上の歪み
- MVPから完成形へ：最初は混ざる前提で、どう“ほどける形”を残すか
- Coding agent が入ったとき、なぜ「なぁなぁ」が許されにくくなるのか
- 中間体としての ROM-rb（Repository Pattern）が効く条件・効かない条件

---

## 1. Railsの価値は「AR」ではなく「統合と規約」
Railsの強みは、アプリ開発に必要な要素（ルーティング、MVC、マイグレーション、ジョブ、設定、テスト、慣習など）を**統合**し、実装の迷いを減らす**規約**を提供する点にある。

- 利点：考えることが少なく、安全に立ち上がる  
  - “こう書けばこうなる”が揃っている  
  - チームの共通言語になる  
  - 実装が散りにくい（少なくとも散りにくい方向に誘導する）

一方で、Railsの“副産物”として **ActiveRecord と scaffold** が大きく普及した。しかし、ここをRailsの本体と捉えると、議論がズレる。

---

## 2. Railsの欠点：fat modelの“標準解”が弱く、流派が乱立しやすい
Railsは「fat controllerを避ける」文化が強く、結果としてモデルに責務が集まりやすい。ところが、fat modelの解消に関しては固定の正解が弱く、次の流派が混在しやすい。

- concern
- query object
- form object / command object
- service object（service class）
- repository（ROM-rb等の仕組み）
- engine / pack分割
- …etc

特に service object は
- 責務が揺れやすい
- 返り値が揺れやすい（Hash返し等）
- Rubyでは静的型付けが標準で強制されないため、破綻が増幅しやすい  
という「成功体験が伝播しやすいのに、劣化もしやすい」性質を持つ。

> 「service classが悪い」ではなく、  
> **“静的型付けが標準で強制されないRubyで、返り値と責務が揺れる抽象を大量生産しやすい”**のが致命傷になりやすい。

---

## 3. ActiveRecordモデルが兼ねがちな役割（なぜ溶けるのか）
Railsの典型運用では、ARモデルが次の役割を抱え込みやすい。

1. 型（っぽい役割）
2. データ層（永続化）
3. ビジネスロジック
4. リポジトリ的役割（scopeやクエリ組立）

この「兼務」は初期に爆速を生むが、成長すると境界が溶け、分割（サービス化）や移行（永続化変更）を難しくする。

---

## 4. Zeitwerk と「型の代替」
Zeitwerkは「静的型」を提供しない。しかし、以下を強くする。

- パス ↔ 定数名の一貫性
- “その名前のクラスはどこか”を追いやすくする規律
- 命名規約に乗った探索性

その結果、Railsでは「クラス名＝概念＝型の代替」として運用しやすい土台ができる。

> Zeitwerkは型検査ではない。  
> しかし **“型としてクラス名を運用する文化”**を成立させる。

---

## 5. 「read と update は別ドメインか？」
結論はこうなる。

- **別ドメイン“であるべき”とは言えない**
- ただし **別コンポーネント（場合により別サービス）になり得る**

特に次の差が出ると分離が自然になる。

- ルール（不変条件）の差：更新は強整合、読みは表示都合
- データ形の差：正規化（write） vs 非正規化（read）
- 整合性要求の差：強整合 vs 結果整合

この分離は「別ドメイン」よりも **同一ドメイン内のread model / write model分離（CQRS的）**として捉えると誤解が少ない。

---

## 6. 新規開発はCRUD解像度で始まる（そして混ざる）
ActiveRecord系フレームワークの新規開発では、初期にこう書くのが自然。

- `xx` をCRUDする

そして現実として、最初は混ざる。

- 読み都合（一覧、検索、並べ替え）が正規モデルに侵入する
- 副作用（通知、索引更新）がafter_commitに入る

「最初から混ぜるな」は現実的でない。重要なのは次。

> **“混ぜてもいい。しかし混ぜ方に上限と出口（ほどける道）を用意する”**

---

## 7. “混ぜる”を前提に、詰まないための最小ルール
### 7.1 読み（Query）の置き場を固定する
- controllerにクエリを散らさない
- scopeを無限増殖させない
- Query（Relationを返す）に集約する

### 7.2 副作用は散らさない
- after_commitに書いてよいのは「イベント記録 or job enqueue」まで
- 外部API呼び出しや索引更新の本体をコールバックに入れない

### 7.3 “分離のトリガ条件”を最初に言語化する
例：
- 検索条件が増えてJOIN地獄になった
- 検索改善が毎日回る一方、取引は慎重にしたい
- 取引スキーマ変更が検索を壊す
- パフォーマンス要求が分岐した  
→ read model（search_documents / index）を導入する

---

## 8. SOA とマイクロサービスの境目（誤解されがちな点）
「ユーザー価値」はしばしば“束”で成立する。

- 取引だけでは価値にならない
- 検索・ブックマーク・通知が揃って体験が成立する

しかしこの“束”と、アーキ境界は一致しない。

### 8.1 マイクロサービスの本質
- 独立デプロイ
- データ所有（他サービスDB直参照をしない）
- 契約で連携（API/イベント）
- 障害/負荷の隔離
- チームが運用まで所有

repoを分けることは本質ではない。  
本質は **自律性**。

### 8.2 Rails/ARの世界観とマイクロサービスの衝突
ARは「同一DBを横断」する快感を提供する。  
マイクロサービスは「データ所有の分離」を要求する。  
この二つは思想として正面衝突する。

そのため、何も手当てせずに Rails way の慣性に乗るほど、マイクロサービス移行の断層は大きくなりやすい。

---

## 9. 「中間体」としての ROM-rb（リポジトリ的アプローチ）
ROM-rbはRepository Patternそのものではない。だが、データアクセスを明示化する中間体として効く領域がある。

### 9.1 ROM-rbが効くこと
- データアクセスの責務を明示化できる
- read model（検索用投影）と相性が良い
- ARの「どこからでも触れる」性質を弱め、境界を作りやすい

### 9.2 ROM-rbが解決しないこと
- DB所有の分離（マイクロサービス本丸）は別問題
- 既存AR依存（クエリ散乱、コールバック散乱）を勝手に剥がしてはくれない

### 9.3 現実的な使い方（例）
- 取引（write）はARで維持
- 検索（read）はROMでread modelを扱う  
→ 将来、検索だけ切り出す道を残しやすい

---

## 10. Coding Agent と協業すると「なぁなぁ」が許されにくくなる
エージェントは実装を加速するが、同時に“散らかり”も加速する。

- 人間なら暗黙知で回っていた例外が、爆速で増殖する
- 設計の揺れが、次の生成品質を下げる
- 結果として一貫性が価値になる

だからエージェント時代は、最初にこれだけ決めるのがROIが高い。

### エージェント協業の“憲法”（最小）
- 返り値の形（AR/Relation/Entity）を固定
- Query置き場を固定（controllerにクエリ禁止）
- コールバックの副作用を制限（enqueueまで）
- 名前空間/命名規約を固定（Zeitwerkで追える）
- 例外を作る場合の手続きを固定

---

## 11. Rubyに型があったらRailsはどうなったか？
Rubyで強い静的型付けが言語標準として強制されていた場合、Railsの“暗黙”は今ほど極端には寄れなかった可能性が高い。
その代わり新しいコストが生まれる。

- 記述量の増加（境界の型注釈）
- どこまで型を付けるか、という設計判断の増加
- メタプロと型の緊張関係（回避策の増殖）

一方で、service objectの破綻（返り値の揺れ）は型で大きく改善される。

---

## 12. 結論：Rails×DDD×Agent の現実的な到達点
- Railsは「統合と規約」の器として強い
- AR中心主義は初期爆速を生むが、成長すると境界が溶ける
- 最初は混ざる前提で、ほどける道（Query/副作用/read model）を残す
- マイクロサービスは“自律性”が本質で、Rails/ARの快感と衝突する
- ROM-rbは中間体として使えるが、魔法ではない
- Coding agent時代は「なぁなぁ」が負債化する速度が上がるため、少数のガードレールが重要

---

## 付録：Railsの整理（短い要約）
**Rails**  
- **利点**：規約と統合により、考えることが少なく安全に立ち上がる（fat controllerになりづらい）  
- **欠点**：ARが兼務しfat modelになりやすい。解消の標準解が弱く、service object / concern / query等の流派が乱立しやすい（特にservice objectは型なしRubyで破綻が増幅しやすい）。
